# SKILL: ポスティング・マップ・システム Pro版の構築と汎用化

## 概要
このスキルは、特定の地域（三重2区等）に依存していたプロトタイプを、全国展開可能なSaaSモデル（Pro版）へと昇格させるための、アーキテクチャ設計と実装手順を定義する。

## コア・アーキテクチャ
1.  **CONFIGによる一括管理**: シート名、ファイル名、動作パラメータを `CONFIG` オブジェクトに集約し、コードの変更なしでカスタマイズを可能にする。
2.  **汎用抽出エンジン**: 選挙区名と都道府県を指定するだけで、CSVから該当エリアを自動抽出するロジック。
3.  **リアルタイム進捗マップ**: Google Maps JavaScript APIを使用し、スプレッドシートの進捗（完了/未完了）を色分けして表示するHTMLダッシュボード。

## 実装手順（原本テンプレート作成）

### 1. CONFIG オブジェクトの定義
スクリプトの冒頭に以下の構造の `CONFIG` を配置し、全関数から参照する。
```javascript
const CONFIG = {
  VERSION: "2.0.0",
  DISTRICT_CSV: "三重県選挙区区割り.csv",
  POSTAL_CSV: "MIE_POSTAL.CSV",
  SHEET_GUIDE: "初めての方「使い方ガイド」",
  SHEET_TEMPLATE: "原本",
  // ...他シート名
};
```

### 2. 選挙区抽出関数の汎用化
`extractDistrictAddresses(district, pref)` 関数を実装し、`districtData`（区割り）と `postalData`（郵便番号）をマージして住所リストを生成する。

### 3. リアルタイム・カラーマップの実装
-   **GAS側**: `getMapData()` で全シートの住所・完了フラグ・担当者をJSONで集計。
-   **HTML側**: `map_dashboard.html` を作成し、ダークモードの地図デザインにピンを立てる。
-   **同期**: `google.script.run` を利用した非同期データ取得。

### 4. 投票率データ連携と戦略的強調
-   **データソース**: `voter_turnout.csv`（市区町村, 町域, 投票率, 優先度, 戦略メモ）を使用。
-   **紐付けロジック**: `getMapData` 関数内で、住所テキストに市区町村＋町域が含まれるかを部分一致で判定。
-   **ビジュアル表現**: 
    -   優先度に応じてピンの `scale` を動的に変更。
    -   最優先（Priority 3）には特別なストロークカラー（黄色）を付与。
    -   InfoWindow内に投票率や戦略情報をリッチに表示。

### 6. SaaS基盤とライセンス認証（Phase 3）
-   **マスター司令塔**: `master_gas.gs` によりStripe Webhook（`checkout.session.completed`）を受信し、顧客DBを自動更新。
-   **ライセンス判定API**: `doGet` エンドポイントを公開し、顧客シートからの認証リクエストにJSONで回答。
-   **クライアント・ガード**: 
    -   `onOpen` 時の自動メールチェック。
    -   `UrlFetchApp` によるマスターへの問い合わせ。
    -   非アクティブ時のメニューロックおよび警告表示。

### 7. 管理者権限とUI
-   `onOpen()` で `Session.getActiveUser().getEmail()` をチェックし、オーナーのみに「★ポスティング管理」メニューを表示。
-   メニューには「🗺 地図で進捗を確認」を最上位に配置。

## 注意事項
-   **APIキーの管理**: Google Maps APIキーは、将来的に管理シートまたはスクリプトプロパティから注入する構成とする。
-   **スマホ最適化**: エリアシートの行の高さは一律 `60` ピクセルとし、3行程度のテキストが収まるように設定する。
-   **保護設定**: スタッフが入力する範囲（完了、枚数、担当、メモ）以外は、自動生成時にロックをかける。

## 復旧・再現コマンド
新しい選挙区でシステムを開始する場合は、`CONFIG.DEFAULT_DISTRICT` を書き換えた後、`generateAreaSheets` を実行する。

### 8. ハイブリッド・戦略マップ（Phase 4）
-   **マクロ表示（ズーム < 15）**: 
    -   google.maps.Circle を使用して町丁目単位の進捗を大きな円で可視化。
    -   進捗率に応じた色分け（緑: 80%~, 黄: 30~80%, 赤: ~30%）。
-   **ミクロ表示（ズーム >= 15）**: 
    -   詳細な個別ピンを表示。
-   **自動切り替え**: 
    -   zoom_changed イベントリスナーにより、setVisible() で表示/非表示を動的に制御。
-   **集計ロジック**: 
    -   GAS側で summary オブジェクトを作成し、町丁目ごとの total と done を事前計算してフロントに送信。

### 9. エグゼクティブ・ダッシュボード（Phase 5）
-   **ビジュアル進捗**: SPARKLINE関数（charttype: bar）を使用して、目標枚数に対する進捗を視覚化。
-   **動的レイアウト**: createTextFinder を活用して、特定のラベル（「更新日時」など）の隣接セルに動的にデータを流し込む。
-   **プロフェッショナル・デザイン**: 背景色（#1a237e / #f8f9fa）、フォントサイズ（24pt）、カラーリングを使い分け、信頼感のあるダッシュボードを構築。
-   **自動保護管理**: ダッシュボードの重要項目を保護しつつ、チェックボックス（トリガー）のみを unprotectedRanges で開放し、操作性と安全性を両立。

### 10. マルチデバイス・戦況共有戦略（KML連携）
-   **KMLエクスポート**: Googleドライブ上にKMLファイルを自動生成し、Googleマイマップへのインポートを可能にする。
-   **視覚的整合性**: KML内の Style タグを使用して、スプレッドシートの「完了（緑）」「未完了（赤）」の状態をマイマップ上でも完全再現。
-   **営業活用**: タブレット（iPad）でのリアルタイムマップ実演と、本部モニター用のマイマップ提案を組み合わせ、高単価案件の成約率を向上させる。

### 11. 大規模データ処理（GAS 6分制限の回避）
-   **バッチエンジン**: PropertiesService を使用して BATCH_INDEX や BATCH_CITY_COUNTS などの実行状態を永続化。
-   **タイムアウト制御**: new Date().getTime() で実行時間を監視し、5分（300秒）を超えた場合に処理を分割。
-   **自動再開トリガー**: ScriptApp.newTrigger を使用して、中断ポイントからの再開を1分後に予約。完了後に deleteTriggers で掃除。
-   **UX**: バッチ実行開始前に ui.alert で所要時間の目安を提示し、安心感を醸成。

### 12. GASパフォーマンス最適化（処理速度の向上）
-   **Rangeの最小化**: setWrap や setBackground などの書式設定は、getMaxRows() で全体に適用せず、データが存在する「必要な範囲」だけに絞って適用する。
-   **API呼び出しの削減**: 1回の setValues でまとめて書き込むなど、Googleサーバーとの通信回数を最小化することで、実行時間を大幅に短縮できる。

### 13. ゼロ・コール原則（徹底的なAPI最適化）
-   **条件付き実行**: setRowHeights や setValues などのAPIを叩く前に、必ずデータ件数（length > 0）をチェックする。これにより、空データ時のエラー回避と処理時間の微減を同時に実現する。
-   **バルク処理の活用**: 個別の row ごとに処理せず、startRow と numRows を指定する「複数行一括操作」系メソッドを優先的に使用する。

### 14. 行操作の軽量化（Hide vs Delete）
-   **表示の最適化**: 空白行を deleteRows で削除すると再計算コストが高いため、hideRows で非表示にする。これにより、ユーザーには「必要な分だけシートがある」というアプリ感を与えつつ、実行速度を劇的に向上させる。
-   **注意点**: セル数制限（1000万セル）を考慮し、テンプレート原本の行数をあらかじめ必要最小限（例：50行程度）に絞っておく運用が最強。

### 15. 処理順序の最適化（リフロー抑制）
-   **シーケンシャル・バッチング**: 同一 Range への操作（setValues, setWrap 等）を隣接して実行する。これにより、スプレッドシート内部の再描画（Reflow）回数を最小化し、大量シート生成時の全体的なスループットを最大化する。

### 16. スケーリング戦略（キャッシュServiceの活用）
-   **CacheServiceの導入**: 計算コストの高い集計処理や、外部ファイル（CSV）の読み込み結果を JSON.stringify してキャッシュ。
-   **100KB制限の回避**: キャッシュ保存前に文字列長を確認し、制限を超える場合はキャッシュせず実処理を行う例外処理を実装。
-   **ユーザー/シート別キー**: スプレッドシートIDやユーザー識別子をキーに含め、マルチテナント（複数顧客）環境下でのキャッシュ汚染を防止。

### 17. Headless GAS & UIの分離（Decoupling）
-   **バックエンドの純粋化**: スプレッドシート側の整形（wrap, height調整）を最小限に抑え、GASは「データの計算と返却」に特化させる。これによりシート生成速度を最大化する。
-   **フロントエンドの高度化**: HTML/JS/CSS側で表示ロジック（改行処理、条件付きカラー、アニメーション）を完結させる。
-   **Web App API**: doGet を実装し JSON を返却することで、GASを汎用的なバックエンドAPIとして活用する。

### 18. 座標永続化（ジオコーディングの資産化）
-   **事前取得ロジック**: 大量データを扱う場合、地図表示時の動的変換を避け、データ生成時（バッチ処理内）に緯度経度を取得してセルに保存（J, K列）する。
-   **APIクォータ管理**: Maps.newGeocoder() は呼び出し制限があるため、バッチ処理と組み合わせることで制限を回避しつつ、確実に全データの座標を確保する。
-   **高速表示（No-Geocoding Map）**: 保存された座標をAPI経由でフロントエンドに渡し、マーカー配置の遅延を解消する。
